# 公众号发布平台构建计划

## 目标
构建一个可长期运营、可迭代、可多人协作的公众号内容发布平台。

## 执行节奏
- 每天 20:00（Asia/Shanghai）自动执行一项任务
- 每周 7 天连续执行
- 从 Phase 1 Day 1 开始
- 若当天任务提前完成，自动继续下一项
- 用户未明确终止前，任务持续进行

## 完整计划

### Phase 1：夯实基础

| Day | 任务 | 交付物 | 验证标准 |
|----|------|--------|---------|
| 1 | 统一配置管理：把 publish_v6 硬编码的 AppID/Secret 迁到 config 表，支持环境变量覆盖 | `modules/publish.js` 可配置化 | 不修改代码就能切换公众号 |
| 2 | 服务进程守护：pm2/systemd 自启 + `/api/health` 健康检查 | systemd 服务文件 / pm2 配置 | `systemctl status` 或 `pm2 list` 显示 running |
| 3 | 数据库迁移化：`schema.js` 改为可 up/down 的迁移脚本 | `migrations/` 目录 | 新加字段不丢数据 |
| 4 | 日志规范化：`pino` 替换 console.log，按天轮转 | 日志文件 | 不再直接刷屏，可检索 |
| 5 | 公式渲染收尾：根据预览效果微调，增加本地预览模式 | `preview/` 本地 HTML | 不发微信也能看最终效果 |
| 6 | 发布模块整合：publish_v6 核心能力并入 server.js 的 publish 模块 | API 端点 | 前端/接口调用即可发布 |
| 7 | 前端最小可用：评估现有 public/index.html，修复或重建关键页面 | 可用 Web UI | 能在网页完成一次发布 |

### Phase 2：内容生产能力

| Day | 任务 | 交付物 | 验证标准 |
|----|------|--------|---------|
| 8 | 多模型接入：抽象 LLM 接口，支持 OpenAI/Kimi/DeepSeek 等 | `modules/llm.js` | 可切换模型生成文章 |
| 9 | Prompt 模板拆分：按科普/论文/教程等类型建立模板 | `prompts/` 目录 | 生成质量可感提升 |
| 10 | 素材增强：arXiv/PDF 解析、网页抓取、关键词搜索 | 素材采集模块 | 输入 URL 可自动入库 |
| 11 | 配图自动化：封面图裁剪、图库搜索、AI 生图 | 图片处理模块 | 封面尺寸正确 |
| 12 | 版本管理：文章多版本与 diff | 版本历史 | 可对比版本差异 |

### Phase 3：发布与运营能力

| Day | 任务 | 交付物 | 验证标准 |
|----|------|--------|---------|
| 13 | 发布工作流：审核通过 → 预览 → 群发/定时/仅草稿 | 完整工作流 API | 5 步内完成发布 |
| 14 | 多账号/多平台：支持多个公众号，预留知乎/头条接口 | 多账号配置 | 可切换公众号发布 |
| 15 | 发布前检查清单：字数、敏感词、链接、图片检查 | 自检模块 | 拦截常见问题 |
| 16 | 发布回执与重试：失败自动重试、限流处理 | 发布日志 | 失败可追踪 |

### Phase 4：数据驱动的优化迭代

| Day | 任务 | 交付物 | 验证标准 |
|----|------|--------|---------|

| 18 | 内容标签与选题评分 | 选题评分 | 选题库可排序 |
| 19 | A/B 测试：标题/封面/发布时间 | A/B 记录 | 可记录实验结果 |
